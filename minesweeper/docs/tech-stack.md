# Detonator 技術スタック仕様書

Detonatorは、ローグライトマインスイーパーパーティーゲームです。10フロアクリア型で、Colyseusによるリモートマルチプレイに対応しています。

## アーキテクチャ全体図

```
+-----------------------------------------------------------+
|                  クライアント (ブラウザ)                     |
|  +-----------------------------------------------------+  |
|  | React 19 (UI: HUD, アイテムスロット, レベルアップ選択) |  |
|  +-----------------------------------------------------+  |
|  | Zustand (Colyseus状態 → React/PixiJSへの橋渡し)        |  |
|  +-----------------------------------------------------+  |
|  | PixiJS + @pixi/react (盤面・キャラ・エフェクト描画)     |  |
|  +-----------------------------------------------------+  |
|  | Howler.js (音声)                                      |  |
|  +-----------------------------------------------------+  |
|  | Bun (パッケージマネージャー) / Vite (ビルドツール)      |  |
|  +-----------------------------------------------------+  |
+-----------------------------------------------------------+
        | WebSocket (差分状態同期)
        v
+-----------------------------------------------------------+
|                     サーバー (Node.js)                      |
|  +-----------------------------------------------------+  |
|  | Colyseus (Room管理, マッチメイキング, 状態同期)         |  |
|  +-----------------------------------------------------+  |
|  | Vanilla TypeScript (ゲームロジック: 移動/爆発/侵食/判定)  |  |
|  | setSimulationInterval = サーバーサイドゲームループ       |  |
|  +-----------------------------------------------------+  |
+-----------------------------------------------------------+
        |
        v
+-----------------------------------------------------------+
|                   設定・データ層                             |
|  JSON (アイテム/スキル/ステージ/数式) + Zod (検証)          |
|  + expr-eval (数式コンパイル)                               |
+-----------------------------------------------------------+

横断的ツール: TypeScript / Vitest + Testing Library / ESLint + Prettier
```

## レイヤー別詳細仕様

### Layer ① 設定・データ層

**役割**: アイテム、スキル、ステージ、数式、バランスパラメーターをコードから分離して管理する

**採用技術**: JSON (実行時データ) + Zod (ビルド時スキーマ検証) + expr-eval (数式の安全なコンパイル・実行)

**採用理由**: JSONはTypeScriptと最も相性が良いデータフォーマットです。Zodによってビルド時に設定ミスを検出できます。expr-evalは数式文字列を安全に実行できるため、バランス調整用の数式をJSON内に記述可能です（例: `baseRate * (1 + floor * 0.1)`）。

**不採用**: YAML（JSONに決定済み）、TOML（ネスト深いデータ構造に不向き）

**AI相性**: JSON + Zodの組み合わせはAIが最も得意とする領域の一つです。

### Layer ② ネットワーク層

**役割**: 複数プレイヤーの入力を集め、ゲーム状態を全員に同期する

**採用技術**: Colyseus（権威あるサーバーパターン）

**採用理由**: ゲーム専用フレームワークでRoom管理・状態同期・マッチメイキングが内蔵されています。@colyseus/schemaによる増分バイナリ同期（差分だけ送信）を実現。setSimulationIntervalでサーバーサイドゲームループを提供。React公式SDKが存在し、TypeScriptネイティブです。

**不採用**: Socket.io（ゲーム機能は全部自前実装が必要）、ネットワーク層なし（リモート対応は最初から必須要件）

**AI相性**: Context7に1400件以上のスニペットが存在。公式Reactドキュメントも充実しています。

**権威あるサーバー構成**: サーバーがゲーム状態を所有し、クライアントは入力を送信・状態更新を受信するのみです。チート対策とロジック集中管理を両立します。

### Layer ③ ゲームシミュレーション層

**役割**: ゲームのルールを司る。移動、地雷判定、爆発・誘爆、侵食、経験値・レベルアップ、アイテムドロップを管理

**採用技術**: Vanilla TypeScript

**採用理由**: このゲームの判定はすべてグリッド上の離散イベントです（連続物理演算不要）。ColyseusのsetSimulationIntervalがゲームループになります。シーン管理は不要です。Colyseus SchemaのgamePhaseフィールド（lobby/playing/floor-clear/level-up/victory/game-over）で状態遷移を管理し、React側でパターンマッチして画面切り替えを行います。

**不採用**: Phaser（物理エンジン不要、1.2MB肥大化、React統合が不自然）、PixiJS（描画専用でシミュレーション層ではない）

**AI相性**: Vanilla TSはAI学習データが最も多い領域です。

### Layer ④ レンダリング層

**役割**: ゲーム状態をユーザーの目に見える形（ピクセル）に変換する

**採用技術**: PixiJS + @pixi/react

**採用理由**: WebGL高速描画を実現。@pixi/reactでReactの宣言的コンポーネントとして記述可能です（Application, pixiContainer, pixiSprite, pixiGraphics, pixiText）。useApplication()でPixiJS直接アクセス、useTick()で毎フレーム更新が可能です。爆発エフェクト・侵食警告にパーティクルやフィルターを活用できます。

**不採用**: Pure React DOM（視覚効果に限界あり）、Konva.js（遅い、コミュニティ小）、Phaser（重すぎる）

**AI相性**: Context7にpixi-react 121件以上のスニペット、pixijs 5807件以上のスニペットが存在します。

### Layer ⑤ 状態管理層

**役割**: ゲームシミュレーションの結果をUI/PixiJSが読み取れる形に渡す

**採用技術**: Zustand

**採用理由**: vanilla API（getState/setState/subscribe）がReactの外で動作します。Colyseusのroom.onStateChange → store.setState()で直接連携可能。selector（useStore(s => s.hp)）で必要な部分だけ購読し無駄な再レンダーを回避します。軽量でボイラープレートが少ないです。

**不採用**: Redux Toolkit（ボイラープレート多、ミドルウェアチェックが重い）、Jotai（スナップショット型管理に不向き）、XState（マクロフェーズ管理のみに有効）

**AI相性**: 豊富な学習データがあり、シンプルなAPIです。

### Layer ⑥ UIフレームワーク層

**役割**: 盤面以外のインターフェース（HUD、アイテムスロット、レベルアップ三択、タイマー、コンボ表示）

**採用技術**: React 19

**採用理由**: 圧倒的にAI学習データが最も多いフレームワークです。宣言的UIで直感的、コンポーネント再利用が容易です。PixiJSキャンバスの上にオーバーレイとして配置します。

**不採用**: Svelte（学習データがReactの数分の一）、SolidJS（コミュニティ小）

**AI相性**: 全フレームワーク中最強です。

### Layer ⑦ ビルドツール層

**役割**: TypeScriptコンパイル、バンドル、開発サーバー、HMR

**採用技術**: Vite

**採用理由**: 爆速HMRでAI駆動開発と相性抜群です。ネイティブESMを使用。React公式テンプレートが存在し、TypeScriptネイティブサポートがあります。

**不採用**: Next.js（SSR不要、オーバーエンジニアリング）、webpack（遅い、複雑）

### Layer ⑧ パッケージマネージャー

**役割**: 依存パッケージのインストール・管理

**採用技術**: Bun

**採用理由**: npmの10-30倍のインストール速度を実現。npm互換package-lock.json出力対応（Bun 1.3以降）。Vite/React/Colyseus/PixiJSとの相性確認済み。先輩の推奨による実質的サポート体制があります。

**不採用**: pnpm（強力な代替だがBunより遅い、ただしディスク効率とモノレポでは優位）、npm（最遅）

**注意点**: ニッチパッケージで稀にインストール失敗の边缘例があります。

### Layer ⑨ オーディオ層

**役割**: 効果音の再生・管理

**採用技術**: Howler.js

**採用理由**: Web Audio APIをデフォルト使用（低レイテンシ）。autoUnlockでブラウザ自動再生ポリシーに対応。プリロード/キャッシュ/オーディオスプライト対応。pool設定で同時再生数調整が可能。自動suspend/resume管理を行います。

**不採用**: Web Audio API直接（低レベルすぎ、AIが書きにくい）、Tone.js（音楽制作寄り、SFXにはオーバーキル）、PixiJS Sound（機能不足）

**注意点**: ブラウザの自動再生ポリシー自体は回避不可です。ゲーム開始時のユーザーアクション（Startボタン等）で初期化が必要です。

## データフロー図

```
プレイヤー入力
    |
    v
Colyseus Client SDK ---- WebSocket ----> Colyseus Server Room
                                              |
                                    setSimulationInterval
                                              |
                                    ゲームロジック更新
                                    (移動/爆発/侵食/判定)
                                              |
                                    @colyseus/schema
                                    差分を自動検知
                                              |
    <--- WebSocket (差分のみ) ------------------
    |
room.onStateChange(newState => store.setState(newState))
    |
    +---> Zustand Store 更新
    |         |
    |         +---> @pixi/react (useTick等) → PixiJS描画更新
    |         |
    |         +---> Reactコンポーネント (selector購読) → HUD/アイテム/レベルアップ再描画
    |
    +---> Howler.js → 効果音再生
```

## 開発フェーズ

**フェーズ1: コアゲームロジック開発 + ローカルテスト**

同一ブラウザで複数タブを起動してテスト可能です。ネットワーク層は最初から設計に組み込みますが、開発初期はローカル環境で単一テストが可能です。

**フェーズ2: マルチプレイ統合**

複数デバイスからの参加を実現します。権威あるサーバー構成により、チート対策とロジック集中管理を両立します。

## 主要設計決定サマリー

| 項目 | 決定 | 理由 |
|------|------|------|
| 設定フォーマット | JSON + Zod | TypeScriptとの親和性、ビルド時検証 |
| マルチプレイ | Colyseus (権威あるサーバー) | 最初からリモート対応 |
| ゲームエンジン | 使用しない (Vanilla TS) | グリッド離散イベントのみ、物理演算不要 |
| レンダリング | PixiJS + @pixi/react | WebGL高速描画、React宣言的統合 |
| 状態管理 | Zustand | React外からの操作可能、selectorで最適な再レンダー |
| UI | React 19 | AI学習データ最強 |
| ビルドツール | Vite | 爆速HMR、AI駆動開発に最適 |
| パッケージマネージャー | Bun | 最速インストール、先輩推奨 |
| 音声 | Howler.js | Web Audioラッパー、ブラウザ互換性 |

## まとめ

このスタックは、AI駆動開発との相性、TypeScriptネイティブサポート、リモートマルチプレイ対応を最優先に設計されています。各レイヤーは明確な責務分離により、開発効率と保守性を両立します。
