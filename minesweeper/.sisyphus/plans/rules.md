# Detonator 開発ルール

> **⚠️ このファイルは参照ドキュメントです。タスク定義やQAシナリオは含みません。**
>
> このファイルは全ドメイン（D1-D10）に共通する開発ルールの集合です。
> **実行用のタスク・QA・手順は各ドメインのプランファイル（`01-*.md` ~ `10-*.md`）に記述されています。**
> 全体の実行順序とドメイン間の依存関係は `00-overview.md` を参照してください。
>
> 各ドメインプランの構造:
> - ステップ1: 依存インストール
> - ステップ2: テストを先に書く（RED）
> - ステップ3: 基盤・インフラ構築
> - ステップ4: 実装（GREEN）
> - ステップ5: テスト通過確認（`bun run test`）
> - ステップ6: 人間目視確認（ブラウザで動作確認 + スクリーンショット保存）

## 1. アーキテクチャルール

### 1.1 権威コアパターン（Authoritative Core Pattern）

- ゲームロジックは**純粋なTypeScriptモジュール**として実装する
- React/Colyseus/PixiJSへの依存を**コア内部に持ち込まない**
- コアAPI: `commands → reducer → events → nextState`
- コアは**ローカルでもサーバーでも全く同じコード**で動く
- Colyseusは**トランスポートアダプタ**として後から接続する（D8で導入）

> **⚠️ D1/D2での簡略化**: D1（プレイアブルマインスイーパー）とD2（PixiJS描画）では、
> コアの完全なcommand/reducer/eventsパイプラインは省略し、UIが`GameState`を直接操作する
> スタイルで実装する。これは**プロトタイプ用の簡略化**であり、D8（Colyseus接続）で
> 本格的なcommand/reducer/eventsパイプラインに再構築される。
> D1/D2では以下の最小限の分割のみを守る:
> - コア関数（`revealCell`, `toggleFlag`, `generateBoard`等）は純粋関数として実装
> - UIはコア関数を呼び出し、戻り値を`setState`で反映
> - コア関数はReact/PixiJSのimportを持たない

### 1.2 決定性（Determinism）

- コア内で`Math.random()`や`Date.now()`を**直接使用しない**
- 乱数・時間は**注入可能なプロバイダ**を通じて取得する
- 同じseed + 同じ入力 → 常に同じ出力（テスト・再現性の前提）
- フレーム依存の計算は`dt`（delta time）を引数で受け取る

### 1.3 レイヤー分離

```
[Core]    ← 純粋TS。ゲームルール・状態・計算。テスト可能。
  ↓ view models / selectors（安定した読み取り専用インターフェース）
[UI]      ← React + PixiJS。描画と入力のみ。コア内部を直接読まない。
[Transport] ← Colyseus（D8以降）。コアへのコマンド送信と状態配信のみ。
```

- UIはコアの**内部状態**を直接読まない。View Model / Selector経由のみ。
- Transportはコアの**コマンドAPI**を呼ぶだけ。ロジックを持たない。

## 2. ゲーム設計定数（interview2確定分）

> 以下はinterview2で確定した設計値。実装中に変更する場合は必ずinterview2を更新すること。

### 2.1 セルタイプ

| セルタイプ | 移動 | 掘削 | 説明 |
|---|---|---|---|
| `safe_cell` | 可能 | 不可 | 掘られたマス。周囲地雷数を表示 |
| `mine_safe` | 不可 | 可能 → safe_cell | 地雷なし。mine_dangerと目視不可 |
| `mine_danger` | 不可 | 可能 → 爆発 | 地雷あり。mine_safeと目視不可 |
| `wasteland` | 可能（減速） | 不可 | 基礎速度×0.4 |
| `hole` | 不可 | 不可 | 初期配置。Bridgeでsafe_cellに変化 |

### 2.2 移動・操作

- **移動方式**: 連続座標（浮動小数点）+ AABB矩形コリジョン
- **基礎速度**: 2セル/秒
- **荒地速度**: 基礎速度 × 0.4
- **速度合成**: 乗算合成 `base × wasteland(0.4) × dash(1.5) × (1 + skillStack)`
- **向き（Facing）**: 最終移動方向固定。Dig/Flagは8方向（45度刻み）、地雷除去機は4方向（90度刻み）
- **Dig/Flag**: 地雷原（mine_safe/mine_danger）にのみ可能。荒地・穴・安全マスには不可
- **プレイヤー同士**: AABBコリジョンあり。押し合い可能。同一セルに複数人立つことは理論上可能（プレイヤーサイズは1マス弱）

### 2.3 距離定義（場面ごとに別々）

| コンテキスト | 距離定義 | 具体値 |
|---|---|---|
| 誘爆範囲（衝撃波）| チェビシェフ | 半径1（周囲8マス）|
| 荒地化範囲（爆風）| マンハッタン | 半径2（縦横2マス、斜め1マス）|
| チェックポイント検知 | ユークリッド | `dist² ≤ R²` |
| 起爆探索（導火線）| チェビシェフ | 旗と旗の間 |
| 侵食警告 | 距離概念ではない | frontline隣接マスが直接的に警告対象 |

### 2.4 ゲーム全体ルール

- **勝利条件**: 10フロアクリアのみ
- **敗北条件**: 全員同時死亡 = 即ゲームオーバー（救済措置なし）
- **プレイヤー人数**: 1~10人（hard cap 10、推奨4~6人）
- **チーム**: 単一チーム（全参加者＝同一チーム）
- **経済システム**: なし（Money/Shop完全撤去）
- **アイテム獲得**: ドロップ + レベルアップ提示のみ
- **和音（chord）**: レアスキルとして1回のみ取得可能。取得後はそのラン中常時有効。
- **爆発連鎖速度**: 1/8秒/セル
- **クライアント予測**: なし。サーバー権威のみ
- **チャット機能**: 未実装

### 2.5 フロア・ステージ

- **フロア** = 進行の単位（マリオの1-1のようなもの）
- **ステージ** = 盤面レイアウト（1フロアにつき複数ステージからランダム選択）
- **フロア遷移時の持ち越し**: スキル・アイテム
- **フロア遷移時のリセット**: 地形・旗・CT・バフ・一時効果・プレイヤー位置
- **休憩フェーズ**: あり（レベルアップ報酬選択時間）
- **クリア演出中**: タイマー停止、地雷原消滅、全員復活、初期スポーン移動

## 3. 開発ワークフロールール

### 3.1 タスク実行順序（各タスク必須）

```
1. 依存インストール    → その機能に必要なパッケージのみbun add
2. テストを先に書く    → RED（失敗するテスト）
3. 基盤・インフラ構築  → 型定義、モジュール構造、インターフェース
4. 実装               → GREEN（テストを通す最小実装）
5. テスト通過確認      → 全テストPASS
6. 人間目視確認        → ブラウザで動作確認（スクリーンショット保存）
```

### 3.2 依存インストールの原則

- **機能ごとに必要な依存だけをインストール**する（全部 upfront ではない）
- 依存インストールは各ドメインの**最初のタスク**で行う
- インストール時は`bun add`を使用し、devDependenciesとdependenciesを正しく分ける
- 各ドメインのプランに「このドメインでインストールする依存」を明記する

### 3.3 各ドメイン完了条件

- そのドメインの**全テストがPASS**している
- **人間がブラウザで動作を確認**している（スクリーンショットを`.sisyphus/evidence/`に保存）
- **コードベースが壊れた状態で残らない**（全ドメインのテストが通る）
- **コミット**が作成されている

## 4. テストルール

### 4.1 テスト戦略

- **コアロジック**: テスト**必須**。各ドメインでユニットテストを作成
- **フロントエンド**: テスト実装あり + **人間目視確認**が主
- **統合テスト**: ドメイン間の結合は適宜統合テストを追加

### 4.2 テスト品質

- **決定的**: 同じ入力 → 常に同じ結果。seed固定で乱数を制御
- **時間依存テスト**: fake timerを使用（`@sinonjs/fake-timers` または `vi.useFakeTimers()`）
- **エッジケース**: 境界値・空入力・異常系を最低1ケース含める
- **テスト名**: 日本語で「〜の場合、〜になること」形式

### 4.3 テストファイル配置

```
src/
  core/
    gameLogic.ts
    gameLogic.test.ts        ← パターンA: 実装と同階層
  systems/
    detonate/
      primMst.ts
      primMst.test.ts
    __tests__/               ← パターンB: __tests__/ ディレクトリに集約
      erosion.test.ts
      chain-explosion.test.ts
```

> どちらのパターンでも可。プロジェクト内で一貫していればよい。
> 小規模モジュール（1ファイルのテスト1〜2個）はパターンA、
> 複数ファイルを横断するテストはパターンBを推奨。

## 5. コード品質ルール

### 5.1 TypeScript

- **strict mode**: 常時有効
- **`as any`禁止**: 型が合わない場合は正しい型を定義する
- **`@ts-ignore`禁止**: コンパイルエラーを隠蔽しない
- **`console.log`禁止**: デバッグ用出力は本番コードに残さない

### 5.2 命名規則

- **意味のある名前**: `data`, `result`, `item`, `temp`等の汎用名を避ける
- **日本語コメント**: コードコメントは日本語で記述する
- **ファイル命名**: kebab-case（`prim-mst.ts`, `game-state.ts`）
  - 例外: ReactコンポーネントファイルはPascalCase（`GameBoard.tsx`, `PixiCanvas.tsx`）

### 5.3 コードサイズ

- 1ファイル = 1責任（300行超えたら分割を検討）
- 1関数 = 1行動（50行超えたら分割を検討）
- 1タスク = 1〜3ファイル（計画段階で指定）

## 6. バージョン管理ルール（jj / jujutsu）

> このプロジェクトは **jj（jujutsu）** を使用してバージョン管理を行う。

### 6.1 コミット粒度

- 各ドメイン完了時に**1コミット**
- 大きなドメインはサブタスク単位でコミットしてもよい
- コミットメッセージ形式: `feat(domain): 簡潔な説明`

### 6.2 定期的なコミット

- **作業中は定期的に `jj commit` を行う**
- 目安: タスク内で意味のある区切りができたらコミット（ファイル単位、機能単位）
- 「今日の作業終わり」には必ずコミットする
- コミット前に **6.3 のチェック** を必ず実行すること

### 6.3 コミット前チェック

- `bun run typecheck` → PASS
- `bun run test` → PASS
- 動作確認済みであること

### 6.4 jjコマンド早見表

```bash
jj commit -m "feat(d1): 盤面生成ロジック"   # コミット
jj log                                      # ログ確認
jj diff                                     # 差分確認
jj status                                   # ステータス確認
```

> 理由: 人間がコードベース全体を把握しやすくするため。

## 8. 禁止事項

- ❌ コアロジック内にReact/Colyseus/PixiJSのimport
- ❌ `Math.random()` / `Date.now()` のコア内直接使用
- ❌ UIコンポーネントからコア内部状態の直接読み取り
- ❌ Transport層でのゲームロジック実行
- ❌ `as any` / `@ts-ignore` / `console.log` の本番残存
- ❌ 機能ごとの依存インストールルールの逸脱（まとめて全部入れる等）
- ❌ テストなしのコアロジック変更
- ❌ 人間目視確認をスキップしたドメイン完了宣言
